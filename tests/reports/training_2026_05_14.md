# Long training run — 2026-05-14

First deliberate training run for the Phase 2 / Task 2.1.2
active-sensing CNN. Goal: see how high IoU can go on the
`DualInputCNN` architecture with a real dataset, modest
regularisation, and ~100 epochs of CPU training. Acts as the
baseline for any future architecture tuning.

## What

`DualInputCNN` (232,337 params) trained on 10 000 stereo
active-sensing samples generated by
`scripts/generate_active_sensing.py`, using AdamW + cosine LR +
weight-decay loop in `src/acoustic_system/learning/train.py`. Eval
on (a) in-distribution data (the same 10 k archive) and (b) a
separately-generated 500-sample held-out archive with a different
RNG seed.

## Why

Establish (a) whether the architecture can learn anything beyond
the smoke-test baseline (val IoU 0.04 on 200 samples / 50 epochs),
(b) whether the "concat fusion" design plateaus quickly, (c) how
big the train/test generalisation gap is for this configuration.

## Configuration

| knob | value | reason |
| --- | --- | --- |
| dataset | 10 000 samples, 64×64 grid, stereo (n_mics=2), spacing=12 cells, 3 random rect obstacles, sizes 4-14 | matches laptop-only 2-mic constraint |
| source | synthetic chirp, fs=200 Hz, sweep 0.02→0.4 Hz over 100 sim-time-units | reproducible without an audio corpus |
| target mask | 64×64 | matches the dataset grid |
| optimiser | AdamW, lr=1e-3, wd=1e-4 | weight decay as the only regulariser besides early stopping |
| schedule | CosineAnnealingLR, 100 epochs | LR decays from 1e-3 to ~0 |
| batch size | 32 | cleaner gradients than smoke test (16) |
| seed | 42 (train), 1234 (dataset), 999 (held-out) | reproducible |
| device | CPU | laptop-only target |

## Takeaways

### What the model learned

The model learned **a strong marginal prior over obstacle
locations** rather than the conditional map from `(sensor, source)`
to obstacle mask. Inspecting `preds_heldout.png`:

- Across all 8 held-out samples, predicted probability mass
  clusters in roughly the same regions (upper-middle, sometimes
  upper-right and centre).
- Predictions do **not** track where the actual obstacles are in
  each held-out scene.
- A few held-out samples score IoU 0.09-0.105 by accident — those
  are scenes where the truth obstacles happen to coincide with the
  model's prior.

This is the classic "easiest-thing-to-learn = marginal not
conditional" failure mode. With 10 k samples, 232 k params, and no
data augmentation, the optimisation finds it easier to fit the
typical-obstacle-location distribution than to discover how to
read `(sensor, source)` features.

### Why the loss curves look the way they do

`loss.png` shows:

- Train loss declines smoothly throughout from ~1.15 to ~0.93.
- **Val loss rises monotonically from ~1.11 to ~1.20** — never
  improves at all relative to its starting value. The model is
  not even briefly generalising; it goes straight to memorising.
- Train IoU rises smoothly from 0 to 0.20 starting around epoch 20.
- Val IoU peaks at 0.042 around epoch 50 and slides back to 0.034
  by epoch 100 (the cosine LR slowdown did not fully arrest the
  overfit).
- LR cosine-decays from 1e-3 to ~2e-7 on a log scale.

### Implications for the next run

Diagnoses:

1. **Capacity vs data**: 232 k params on 10 k same-distribution
   samples is enough to memorise. Either reduce capacity or
   substantially increase data diversity.
2. **No data augmentation**: every sample shows the model
   exactly one (sensor, source, room) tuple. Per-channel gain
   jitter, additive noise, or random time shifts would force the
   encoder to attend to relative timing rather than absolute
   waveform.
3. **Single-pose sample**: with only one sensor pair + one driver
   per room, the inverse is severely under-determined. Multi-pose
   aggregation (Task 2.1.4 in `PROJECT_PLAN.md`) is the
   inductive-bias lever that fits the laptop-only hardware
   constraint without adding mics.
4. **Architecture**: the concat-fusion decoder may not have a
   strong enough conditioning on `(sensor, source)`. A U-Net-style
   skip path, or a cross-attention fusion, would force the
   decoder to consume the input features more directly. Lower
   priority than (1)-(3).

Concrete next experiment (when picked up):

- 50 k training samples with varied obstacle counts (1-6),
  varied driver positions per room, varied mic spacings.
- Same architecture + dropout 0.1 in encoder.
- Data aug: per-channel gain in [0.7, 1.3], additive Gaussian
  noise with σ ≈ 0.01.
- 100 epochs.
- If val IoU still does not move, switch to a multi-pose dataset
  (Task 2.1.4) before trying architecture changes — the under-
  determination problem will not be solved by a bigger model.
